# Phase 4 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none — custom components only |
| Icon library | inline SVG (matches existing pattern in MuteButton.tsx) |
| Font | system-ui (Tailwind default — no custom font declared in codebase) |

**Rationale:** No `components.json` exists. The project uses Tailwind CSS v4 (CSS-first, no config file) with raw utility classes and inline SVG icons. Introducing shadcn for two form screens would be disproportionate. Custom components consistent with the existing pattern are the correct approach. Source: codebase scan + CONTEXT.md D-02.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline gap between icon and label, hint text margin |
| sm | 8px | Spacing between stacked form elements (gap-2 in existing components) |
| md | 16px | Card horizontal padding, field internal padding, right-4 pattern from MuteButton |
| lg | 24px | Card vertical padding, bottom-6 pattern from MuteButton, section breaks |
| xl | 32px | Card max-width inner gutter, vertical space between card and link toggle |
| 2xl | 48px | Vertical breathing room above/below card on taller viewports |
| 3xl | 64px | Not used in this phase |

Exceptions:
- Touch targets on all interactive elements: minimum 44px height (established pattern from MuteButton: `min-h-[44px]`)
- Submit button: minimum 48px height for primary CTA prominence on mobile

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px (text-base) | 400 (font-normal) | 1.5 |
| Label / hint | 14px (text-sm) | 400 (font-normal) | 1.5 |
| Heading (card title) | 20px (text-xl) | 600 (font-semibold) | 1.2 |
| Error inline | 14px (text-sm) | 600 (font-semibold) | 1.4 |

**Rationale:** Existing components use `text-base` (16px) for primary content and `text-sm` (14px) for secondary. `font-semibold` is reserved for error/emphasis states (established in FeedContainer error state). No display size needed — auth screens have no hero copy. Source: codebase scan of FeedContainer.tsx, EndOfFeedSlide.tsx.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#000000` (bg-black) | Page background behind card, full-bleed dark canvas |
| Secondary (30%) | `#1a1a1a` (approx. zinc-900 / `bg-zinc-900`) | Card surface — the centered form card sits on this |
| Accent (10%) | `#ffffff` (text-white) | Primary CTA button background, focused input border, active link color |
| Destructive | `#ef4444` (red-500) | Inline field error text only |

Accent reserved for:
1. Submit button background (Login / Create account)
2. Focused input border ring
3. Link toggle text ("Register" / "Log in" anchors)

**Rationale:** CONTEXT.md D-02 specifies "dark background with light card, matches existing dark video player aesthetic." The existing feed uses pure `bg-black` as the dominant surface. White (`#ffffff`) as accent maintains visual continuity — it is already the primary content color in the feed. Red-500 is the conventional destructive/error semantic color in Tailwind; used exclusively for inline field errors per D-06. Source: CONTEXT.md D-01, D-02; codebase FeedContainer.tsx.

## Interaction Contract

### Floating Label Input (D-03)
- Label starts centered inside the input field at body size (16px, weight 400, color white/50)
- On focus OR when value is non-empty: label translates up to top edge of input, scales to 12px, color white/70
- Transition: `transition-all duration-150`
- Input background: `bg-white/5` (subtle differentiation from card)
- Input border: `border border-white/20` at rest, `border-white` on focus
- Input text: `text-white` at 16px weight 400
- Input padding: `px-4 pt-5 pb-2` (top padding accommodates floated label)
- Input height: 56px minimum (ensures 44px touch area with padding)

### Submit Button (D-08)
- At rest: `bg-white text-black font-semibold text-base` full-width, height 48px, rounded-lg
- On submit (pending): button disabled, spinner replaces label text
- Spinner: white SVG on black or black SVG on white button — use animated spin (`animate-spin`) on an SVG circle arc, 20px diameter
- Disabled state: `opacity-60 cursor-not-allowed`

### Inline Error (D-06)
- Appears immediately below the relevant field
- `text-sm font-semibold text-red-500`
- Margin: `mt-1` (4px) between field bottom and error text
- No icon — text only
- Specific error strings: see Copywriting Contract

### Auth Toggle Link (D-05)
- Positioned below the submit button, centered
- `text-sm text-white/60` for "Don't have an account?" prefix
- `text-sm text-white underline` for the action link ("Register" / "Log in")
- Gap between card bottom and toggle: `mt-6` (24px)

### Route Guard / Redirect (D-09)
- ProtectedRoute checks auth state synchronously from Zustand store
- If unauthenticated: immediate `<Navigate to="/login" replace />` — no render, no flash
- If auth state is loading (initial hydration): render `null` or a black screen, never the feed

### Session Expiry Redirect (D-13)
- On any 401 response from the API: Zustand auth state is cleared, router pushes to /login
- No toast, no modal, no "session expired" message — silent redirect only

## Copywriting Contract

| Element | Copy |
|---------|------|
| Login page heading | Log in |
| Register page heading | Create account |
| Primary CTA — login | Log in |
| Primary CTA — register | Create account |
| Email field label | Email |
| Password field label | Password |
| Password hint text | 8+ characters |
| Auth toggle — on login page | Don't have an account? Register |
| Auth toggle — on register page | Already have an account? Log in |
| Empty state (no data) | Not applicable — auth screens have no empty state |
| Error — email already registered | An account with this email already exists |
| Error — invalid credentials | Incorrect email or password |
| Error — password too short | Password must be at least 8 characters |
| Error — network/server failure | Something went wrong. Please try again. |
| Destructive actions | None in this phase — no account deletion, no data removal |

**Copy decisions:**
- "Log in" (two words, lowercase "in") — matches web convention, avoids "Login" as a noun
- "Create account" — imperative verb, no "Sign up" which sounds less intentional
- "Incorrect email or password" — deliberately ambiguous to avoid username enumeration
- "An account with this email already exists" — specific because enumeration on registration is acceptable (no password confirmed yet)
- Source: CONTEXT.md D-04, D-06, D-10, D-11

- Card is vertically centered: `min-h-dvh flex items-center justify-center`
- On tall viewports (> 800px), card floats at vertical center — no stretching

## Accessibility

| Requirement | Implementation |
|-------------|---------------|
| Input labels | `<label>` element associated via `htmlFor` / `id` pair — floating label is a real `<label>`, not a placeholder |
| Error association | `aria-describedby` on input pointing to error element `id` |
| Error announcement | `role="alert"` on InlineError container — screen reader announces on mount |
| Button state | `aria-disabled="true"` + `disabled` attribute during pending state |
| Focus order | DOM order matches visual order — no tabIndex manipulation needed |
| Password field | `type="password"` — browser handles masking and autofill |
| Minimum touch targets | 44px minimum on all interactive elements (established pattern) |
| Color contrast | White on zinc-900 card: passes WCAG AA (contrast ratio > 7:1); red-500 on zinc-900: passes WCAG AA |
